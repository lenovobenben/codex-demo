# codex-demo

这是一个用于练习 **Codex 协作开发流程** 的演示仓库。

项目定位为**多语言 demo 集合**，当前主要语言方向：**Java / Go / Python**。

> 目标：先把项目基础约定定清楚，再逐步添加小型 demo。


## 0. 当前语言方向

- Java：基础工程脚手架与常见后端场景 demo
- Go：经典算法题与轻量服务 demo
- Python：脚本自动化、数据处理与工具类 demo

## 1. 项目定位

- 这是一个实验性质仓库，代码以「小而完整」的 demo 为主。
- 每个 demo 尽量独立，便于快速验证想法。
- 优先保证可读性与可复现性，而不是过度工程化。

## 2. 推荐目录结构

```txt
.
├── README.md               # 项目说明与使用指南（本文件）
├── demos/                  # 各类 demo（按主题拆分子目录）
│   ├── web/
│   ├── cli/
│   └── data/
├── docs/                   # 补充文档（设计记录、踩坑记录）
└── scripts/                # 辅助脚本（启动、检查、清理）
```

如果你暂时还不确定方向，可以先只创建 `demos/`，后续再按需要细分。

## 3. 初始化建议（适合第一次开始）

1. **先写一个最小 demo**
   - 比如：`demos/hello-codex/`，只包含一个可运行文件和一个简短说明。
2. **每个 demo 配一个 README**
   - 说明「做什么」「怎么跑」「预期输出」。
3. **固定提交节奏**
   - 一个功能点一个 commit，便于回退和复盘。
4. **记录决策**
   - 在 `docs/` 写简短笔记：为什么选这个方案、遇到什么问题。

## 4. README 推荐模板（可复制到每个 demo）

~~~md
# Demo 名称

## 功能
一句话描述这个 demo 做什么。

## 运行方式
```bash
# 示例命令
```

## 预期结果
- 输出/页面效果是什么
- 成功判定标准是什么

## 后续改进
- 可选优化点 1
- 可选优化点 2
~~~

## 5. 开发约定（建议）

- 命名统一：目录和文件尽量语义化。
- 小步提交：每次 commit 只做一件事。
- 先跑后提：提交前至少执行一次对应 demo 的运行命令。
- 文档同步：有行为变化就更新文档说明。
